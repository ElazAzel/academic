## Продуктовый контекст и миссия

AI Strategic Academy — это закрытая образовательная платформа для одной конкретной академии.

- Это **не** публичный маркетплейс курсов.
- Это **не** универсальный LMS-шаблон общего назначения.
- Это **не** публичная библиотека контента.

AI Strategic Academy — это операционная академическая платформа, которая управляет полным жизненным циклом обучения:
- контролируемый доступ (ограничен максимум 2 параллельными сессиями устройств на пользователя);
- пользователи и роли (Admin, Instructor, Student, Curator, Super Curator, Customer Observer);
- курсы (включая структуру Модуль → Блок → Урок);
- модули, блоки и уроки (внутри единого плеера);
- тесты и домашние задания (интегрированные в контекст урока, начисляющие XP и влияющие на прогресс);
- прогресс студентов (последовательное прохождение с жесткой разблокировкой уроков);
- когорты (потоки обучения, дедлайны по модулям, привязка к проектам);
- кураторская поддержка (назначение кураторов, SLA очередей, перенаправление вопросов);
- риски (автоматическое выявление отстающих по факторам неактивности и дедлайнов);
- отчеты и аналитика (интерактивный конструктор отчетов с ролевым разграничением доступа);
- уведомления (SSE в реальном времени, пуши Firebase, SMTP-email);
- сертификаты (идемпотентная автоматическая выдача по факту прохождения >85%);
- публичная верификация выданных сертификатов.

Основной продуктовый принцип:
**«Один путь студента. Один контекст курса. Одна операционная правда.»**

Слушатель не должен чувствовать, что обучение разрознено по разным страницам. Весь процесс прохождения собран в единой цепочке:
`Курс → Модуль → Блок → Урок → Материалы / Тест / Задание / Вопрос / Рейтинг / Завершение`

## 3. Ролевая модель платформы

Платформа жестко разграничивает видимость данных и действия для шести ролей:

### 1. Администратор (Admin)
Управляет всей академией.
- Управление курсами (Course Builder), модулями, уроками.
- Управление пользователями, ролями, доступами и когортами (потоками).
- Назначение кураторов, управление зачислениями, генерация инвайтов.
- Просмотр глобальной аналитики и построение отчетов через интерактивный `ReportDesigner`.
- Просмотр системного лога аудита.
- Ручная выдача и отзыв сертификатов.

### 2. Преподаватель (Instructor)
Владеет образовательным контентом и экспертной частью.
- Управление контентом своих курсов, модулей, блоков и уроков.
- Создание тестов, домашних заданий, загрузка видео и методических материалов.
- Проверка домашних заданий и ведение оценочной ведомости.
- Ответы на экспертные вопросы студентов, перенаправленные кураторами.
- Просмотр аналитики успеваемости по своим курсам и создание отчетов через `ReportDesigner`.

### 3. Слушатель / Студент (Student)
Проходит обучение.
- Главная панель `/student` содержит виджет «Продолжить обучение» (агрегатор последнего урока, активных дедлайнов и ответов куратора).
- **Внимание:** Страница отчетов студента `/student/reports` полностью скрыта и удалена для разгрузки интерфейса.
- Прохождение уроков внутри единого плеера (просмотр видео, чтение, тесты, домашние задания, отправка вопросов куратору прямо из плеера).
- Участие в геймификации: получение XP (+50 за урок, +30 за успешный тест, +40 за домашнее задание, +5 за попытку теста).
- Просмотр интерактивных Sonner-уведомлений и эффектов Confetti при знаковых учебных вехах.
- Отслеживание персональных дедлайнов и динамического календаря учебной активности.
- Скачивание PDF-сертификата после успешного прохождения курса (>85% прогресса, финальное задание сдано).

### 4. Куратор (Curator)
Оказывает оперативную поддержку закрепленным студентам и предотвращает их отчисление (dropout).
- Панель закрепленных студентов, SLA-очередь вопросов и среднее время ответа.
- Ответы на вопросы студентов по урокам, перенаправление сложных вопросов преподавателям.
- Проверка домашних заданий закрепленных студентов.
- Мониторинг учебных рисков (отсутствие в сети более 3 дней, просроченные дедлайны по модулям).
- Ведение заметок по студентам и отправка оперативных push-уведомлений.
- Формирование отчетов по закрепленным когортам через `ReportDesigner`.

### 5. Супер-куратор (Super Curator)
Управляет процессами кураторской поддержки.
- Панель мониторинга нагрузки и SLA всех кураторов.
- Распределение и перераспределение студентов между кураторами.
- Контроль нераспределенных студентов и глобальных рисков когорт.
- Формирование сводных операционных отчетов по кураторской службе.

### 6. Заказчик / Наблюдатель (Customer Observer)
Внешняя read-only роль для контроля обучения со стороны клиента (например, спонсора тендера).
- Мониторинг прогресса участников своего проекта/когорты.
- Просмотр выданных сертификатов в рамках разрешенного объема.
- Экспорт сводных отчетов через `ReportDesigner`.
- **Никогда** не может изменять учебный контент, профили пользователей, выставлять оценки или влиять на ход обучения.
- Доступ строго ограничен с помощью `ALLOWED_ROLES_MAP` (предотвращает ошибки `403 Forbidden` на бэкенде).

## 4. Контроль доступа и безопасность

### Лимит сессий устройств
Платформа строго ограничивает одновременную авторизацию: **не более 2 активных устройств на одного пользователя**.
- Контроль осуществляется через выделенную модель `AuthDeviceSession`.
- При авторизации на 3-м устройстве самая старая сессия автоматически отзывается с признаком `device_limit_exceeded`.
- В журнал аудита пишется событие `auth.device_limit_exceeded`.
- Пользователь на отозванном устройстве автоматически разлогинивается при следующем heartbeat-запросе и перенаправляется на `/login?reason=device-limit` с русскоязычным предупреждением о недопустимости передачи аккаунта третьим лицам.

### Content Security Policy (CSP)
- CSP-заголовки устанавливаются динамически per-request в middleware (`proxy.ts`) с генерацией уникального `nonce` через `crypto.randomUUID()`.
- Nonce пробрасывается в `layout.tsx` через request-headers, позволяя Next.js 16 автоматически инжектировать его во все скрипты гидратации, исключая `'unsafe-inline'` в production.
- В dev-режиме для поддержки HMR/Turbopack разрешен `'unsafe-eval'`.

## 5. UX/UI и визуальные стандарты

Платформа выполнена в соответствии с визуальными стандартами 2026 года:
- Светлая чистая тема (основной фон `#F7F8FA`, поверхности `#FFFFFF`, скругления 16–24px, мягкие тени).
- Полная адаптивность: карточки на мобильных устройствах перестраиваются в 2x2 grid (4 колонки на десктопе) для исключения пустых зон.
- **Safe Area Notch:** Заголовки, сайдбары и мобильное меню адаптированы под вырезы безрамочных смартфонов (`env(safe-area-inset-top)`).
- На мобильных устройствах все широкие таблицы (`Table`) динамически перестраиваются в карточки с переносом текста, чтобы исключить горизонтальный скролл.
- Используются только проверенные и интегрированные библиотеки:
  - `vaul` — для высокопроизводительных нативных мобильных drawer-панелей.
  - `recharts` — для красивых интерактивных графиков в кабинетах кураторов, админов и заказчиков.
  - `canvas-confetti` — для анимации завершения уроков/модулей/курсов.
  - `hls.js` — для адаптивного потокового воспроизведения HLS-видео.
  - `sharp` — для автоматического сжатия и оптимизации изображений на сервере.

## 6. Логика прохождения и геймификация

### Сквозной плеер урока
Слушатель проходит обучение в едином интерфейсе `LessonPlayerShell`. Внутри урока встроены: видеоплеер, текстовые материалы, файлы для скачивания, встроенный тест, блок отправки домашнего задания, форма вопроса куратору и рейтинг урока. Никаких переходов на внешние страницы во время прохождения не допускается.

### Система опыта (XP) и микро-ачивок
Любая активность вознаграждается:
- Прохождение урока: **+50 XP**
- Успешный тест: **+30 XP**
- Сдача домашнего задания куратору: **+40 XP**
- Каждая попытка прохождения теста (мотивационный бонус): **+5 XP**
Студент мгновенно видит начисление опыта через всплывающие Sonner-уведомления. При завершении знаковых этапов запускаются Confetti-эффекты.

## 7. Сертификаты и отчеты

### Идемпотентная выдача сертификатов
Студент может запросить выдачу сертификата из карточки пройденного курса. Система автоматически и безопасно проверяет зачисление, прогресс (>85%) и сдачу заданий. Повторный запрос сертификата работает идемпотентно: он возвращает существующую запись вместо создания дубликатов или генерации ошибок БД.

### Ролевой конструктор отчетов
Отчеты строятся через интерактивный `ReportDesigner` во всех кабинетах.
- Доступные конфигурации отчетов фильтруются на клиенте согласно маппингу разрешений `ALLOWED_ROLES_MAP`, предотвращая падения API с ошибкой `403`.
- Row-Level Security жестко контролирует, чтобы никто не мог скачать данные, выходящие за пределы его зоны ответственности.
- Доступны форматы выгрузки CSV, Excel (XLSX) и PDF.

## 8. Модель уведомлений

- Основной канал по умолчанию — `in_app` с выводом сообщений в реальном времени через Server-Sent Events (SSE).
- SMTP-email отправка включается глобальным флагом `FEATURE_EMAIL_NOTIFICATIONS` и используется только для значимых писем.
- **Исключение для безопасности:** Уведомления о критических событиях безопасности (`device_limit_exceeded`, `password_changed`, `profile_updated`, `certificate_revoked`) **игнорируют** любые пользовательские настройки уведомлений. Они обязаны записываться в базу данных, отправляться in-app и фиксироваться в системном аудите.

## 9. Технические правила разработки (Hard Rules)

### 1. Архитектурная изоляция базы данных
**Категорически запрещено вызывать Prisma Client напрямую из файлов страниц (`/app/**/page.tsx`).**
- Все обращения к БД должны быть инкапсулированы в сервисы бизнес-логики (`server/modules/**/*.ts`) или Server Actions (`server/actions/**/*.ts`).
- Страницы импортируют только сервисные методы или вызывают Server Actions.

### 2. Запрещенные паттерны кода (CI Banned Patterns)
Проект содержит строгие проверки, выполняемые автоматически перед сборкой. Запрещено использовать:
- debugger;
- `@ts-ignore` (используйте `@ts-expect-error` с четким обоснованием);
- хардкод секретов, токенов или паролей к БД;
- английские строки в пользовательском интерфейсе.
Перед коммитом всегда запускайте:
```bash
npm run verify
```

### 3. Автоматизированные тесты и доступность (Accessibility)
- Изменения в верстке или логике маршрутов должны проверяться тестами доступности axe-core (`tests/e2e/accessibility-smoke.spec.ts`) на соответствие стандартам WCAG 2.0/2.1 AA.
- Стабильность E2E тестов: при переходах по страницам запрещено использовать ожидания `networkidle` в Playwright (SSE-стримы держат сеть активной, что приводит к таймаутам). Используйте явные локаторы видимости элементов.

### 4. Версионирование VAPID-ключей
Ключи для push-уведомлений Web Push (VAPID) должны быть абсолютно идентичны в серверных (`VAPID_PUBLIC_KEY`) и клиентских (`NEXT_PUBLIC_VAPID_PUBLIC_KEY`) переменных окружения.
