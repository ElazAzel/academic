Этот документ содержит архитектурный обзор, ключевые пользовательские сценарии, критические инварианты, граф зависимостей и описание сред развертывания платформы **AI Strategic Academy** по состоянию на май 2026 года.

## 2. Ключевые пользовательские сценарии (Top 5 Flows)

1. **Последовательное обучение студента (Sequential Study Flow):**
   Студент заходит в `/student` -> нажимает "Продолжить обучение" -> попадает в плеер урока (`/student/lessons/[lessonId]`). Если режим курса последовательный, следующий урок заблокирован, пока текущий обязательный урок не будет завершен (через успешную сдачу теста, принятие задания куратором или клик по кнопке завершения).

2. **Кураторское сопровождение и проверка работ (Curator Grading & Support Flow):**
   Студент отправляет практическое задание из плеера урока -> Создается запись в `assignment_submissions` со статусом `submitted` -> Куратор видит работу в `/curator/assignments` -> Проверяет её, пишет комментарии и ставит статус `accepted` или `needs_revision`. Студент получает PWA Push-уведомление / In-App оповещение о результате проверки.

3. **Тестирование и автоматический зачет (Quiz Passage Flow):**
   Студент открывает тест (`/student/quizzes/[quizId]`) -> Отправляет ответы -> Вызывается API `/api/v1/quizzes/[quizId]/attempts` -> Проверяется количество попыток и лимиты rate-limit -> Вычисляется процент верных ответов -> Если процент $\ge$ порога прохождения, попытка сохраняется как успешная, обновляется `lesson_progress` и синхронизируется общий прогресс по модулю и курсу.

4. **Автоматическая генерация и верификация сертификата (Certificate Issuance Flow):**
   Студент завершает последний урок курса -> Сервис `progress-service` фиксирует завершение курса -> Генерируется уникальный UUID сертификата -> Создается запись в `certificates` -> Генерируется PDF-документ (используя `pdf-lib` и SVG-шаблон) -> Загружается в S3/MinIO -> Студент скачивает сертификат из `/student/certificates`. Внешний проверяющий может верифицировать подлинность по коду на `/certificates/verify/[verificationCode]`.

5. **Мониторинг рисков и операционная балансировка (Risk & Workload Balancing Flow):**
   Крон-задача (`/api/v1/cron/risks`) анализирует активность студентов -> Если студент не заходил в систему дольше $N$ дней или пропустил дедлайн модуля, в таблице `risk_flags` создается флаг -> Куратор видит риски в `/curator/risks`. Супер-куратор видит общую нагрузку кураторов в `/super-curator/curators` и может перераспределить студентов между ними.

## 3. Критические инварианты (Must Never Break)

* **RBAC & Data Isolation:** Пользователь с ролью `student` ни при каких условиях не должен иметь доступа к кабинетам `/admin`, `/instructor`, `/curator`, `/super-curator` или `/customer-observer`. Server Actions должны вызывать `requireRole` или `requirePermission` на первой строчке выполнения.
* **Изоляция Наблюдателя (Observer Scope):** Пользователь с ролью `customer_observer` видит информацию о прогрессе только тех студентов, которые находятся в закрепленных за ним проектах/потоках. При отсутствии закрепленных проектов система должна возвращать пустой массив, исключая утечку данных о пользователях других клиентов.
* **Последовательность прохождения курсов:** При включенном `sequential` прохождении студент не может открыть $N$-й урок, если не завершены все предыдущие обязательные уроки модуля/курса.
* **Лимиты попыток тестов:** Студент не может совершить попытку тестирования, если исчерпан лимит попыток (`maxAttempts`), заданный преподавателем.
* **Rate Limiting:** Защита от брутфорса авторизации, сброса паролей и спама попыток тестирования через Redis-лимитеры должна функционировать стабильно.

## 4. Граф зависимостей и рискованные зоны (Blast Radius)

* **Auth & Session Management (`server/auth/`):** Изменение логики сессий или Auth.js затронет абсолютно всех пользователей (4000+ человек). Риск блокировки доступа.
* **Служба прогресса (`server/modules/progress/`):** Отвечает за вычисление и запись прогресса уроков, модулей и курсов. Ошибки здесь приведут к зависанию обучения, невозможности сдать курс и сбоям генерации сертификатов.
* **Служба интеграции медиа / S3 (`lib/storage.ts`):** Отвечает за загрузку файлов решений, видео-материалов и сертификатов. Неправильные CORS или права доступа на бакет приведут к невозможности сдать работы или скачать сертификаты.

## 5. Среды разработки и конвейер развертывания (Environments & Deployment)

### Среды:
1. **Local Development (Локальная среда):**
   * БД: PostgreSQL внутри Docker (доступ через port-forward).
   * Кэш: Redis внутри Docker.
   * Хранилище: MinIO (S3-совместимое) локально.
2. **Staging (Тестовая среда):**
   * Полная копия production для проведения интеграционных и E2E тестов.
3. **Production (Боевая среда):**
   * Managed PostgreSQL СУБД с настроенными индексами.
   * Upstash Redis (Serverless).
   * Yandex Cloud Object Storage / AWS S3 для статики и медиа.
   * Хостинг: Vercel или выделенный VPS (через Docker Compose / Kubernetes).

### Конвейер валидации релиза (Release Verification):
Перед каждым слиянием в основную ветку или деплоем запускается обязательная проверка:
```bash
npm run verify:release
```
Она последовательно выполняет:
1. `eslint . --max-warnings=0` (строгий линтинг без предупреждений)
2. `tsc --noEmit` (проверка типов TypeScript)
3. `vitest run` (запуск 354+ unit/integration тестов)
4. `npx prisma validate` (валидация схемы СУБД)
5. `npm run db:generate` (генерация Prisma Client)
6. `next build` (Production сборка приложения)
7. `playwright test` (сквозные E2E сценарии)
