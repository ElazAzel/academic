# Реестр внешних сервисов и технических поставщиков

**Версия:** 1.0  
**Дата:** 01.03.2026  
**Статус:** draft  
**Ответственный за ведение:** Каримова Аружан, руководитель

> **Важно:** Документ является шаблоном и требует проверки юристом с учётом фактической модели обработки данных, договоров, юрисдикции хранения данных и применимого законодательства Республики Казахстан.

---

## 1. Назначение

Настоящий Реестр фиксирует:
- какие внешние сервисы и технические поставщики используются Платформой AI Strategic Academy;
- какие данные передаются каждому сервису;
- где данные хранятся;
- какие риски связаны с использованием каждого сервиса;
- какие меры контроля применяются.

Цель Реестра — обеспечить прозрачность использования внешних сервисов и контроль за обработкой данных.

---

## 2. Реестр сервисов

| № | Сервис | Назначение | Какие данные | Возможное место хранения | Риск | Меры контроля | Статус |
|---|---|---|---|---|---|---|---|
| 1 | **Vercel Inc.** (vercel.com) | Хостинг Next.js-приложения, CDN, Serverless Functions, Edge Functions | IP-адреса, логи запросов, идентификаторы сессий (обезличенные), технические логи | США, Европа (в зависимости от региона) | Средний — логи запросов могут содержать IP и user-agent | HTTPS, выбор региона размещения, ограничение сбора логов | Active (без DPA) |
| 2 | **Supabase Inc.** (supabase.com) | База данных PostgreSQL, объектное хранилище (Storage), авторизация, резервное копирование, realtime | Все данные Платформы: регистрационные данные, учебные данные, файлы, сертификаты, логи, audit logs, резервные копии | США (по умолчанию), Европа (доступны регионы) | Высокий — центральное хранилище всех данных | Private storage, signed URLs, row-level security (RLS), шифрование в покое и при передаче | Active (без DPA) |
| 3 | **GitHub Inc.** (github.com) | Хранение и управление исходным кодом | Исходный код (не содержит реальных данных пользователей, секреты в .env) | США, Европа | Низкий — код не содержит данных пользователей | .gitignore для .env, секретов и данных; приватный репозиторий | Active |
| 4 | **YouTube LLC / Google LLC** (youtube.com) | Отображение видеоматериалов (если используется embedded video) | IP-адрес, cookie YouTube, данные о просмотре видео | США | Средний — cookie и данные просмотра передаются YouTube | Использование youtube-nocookie.com (при возможности) или privacy-enhanced режим | Planned |

## 3. Чек-лист перед подключением нового сервиса

Перед подключением любого нового внешнего сервиса необходимо ответить на следующие вопросы:

| № | Вопрос | Комментарий |
|---|---|---|
| 1 | Нужен ли этот сервис для работы Платформы? | Если нет — не подключать |
| 2 | Какие данные будут передаваться сервису? | Составить список |
| 3 | Есть ли альтернатива без передачи данных? | Например, self-hosted решение |
| 4 | Заключён ли договор/приняты ли условия сервиса? | DPA, Terms of Service |
| 5 | Где хранятся данные сервиса? | Страна, регион, дата-центр |
| 6 | Можно ли удалить данные из сервиса? | Процедура удаления, сроки |
| 7 | Кто имеет доступ к данным на стороне сервиса? | Сотрудники провайдера, субподрядчики |
| 8 | Как отключить сервис и выгрузить данные? | Процедура exit |
| 9 | Есть ли риск трансграничной передачи данных? | Юрисдикция, законодательство |
| 10 | Соответствует ли сервис требованиям безопасности? | Сертификаты (SOC 2, ISO 27001), шифрование |
| 11 | Есть ли у сервиса политика уведомления об инцидентах? | Incident response |

## 4. Статусы сервисов

| Статус | Значение |
|---|---|
| **Active** | Сервис используется в настоящее время |
| **Planned** | Сервис планируется к подключению |
| **Disabled** | Сервис отключён или не используется |
| **Under Review** | Сервис проходит проверку |

---

## 5. Обновление реестра

5.1. Реестр обновляется:
- при подключении нового сервиса;
- при изменении условий использования текущего сервиса;
- при обнаружении новых рисков;
- не реже одного раза в год.

## Места для проверки юристом

- [ ] Актуальность перечня сервисов на момент запуска

- [ ] Юрисдикция хранения данных для каждого сервиса
- [ ] Возможность удаления данных из каждого сервиса

- [ ] Оценка рисков для каждого сервиса
- [ ] Наличие плана exit для каждого сервиса
